## 실험 1: 베이스라인 검색 성능 비교 (1주차)

### 가설
임베딩(벡터) 검색이 일반인 구어체 질문 → 법률 전문용어 간 어휘 격차를 의미론적으로 메워주므로 BM25(키워드) 검색보다 높은 Recall을 보일 것이다.

### 방법
- 동일한 2929개 청크 인덱스에 대해 세 가지 검색기 비교
- BM25: kiwipiepy 형태소 분석 기반 한국어 키워드 검색
- 임베딩: jhgan/ko-sroberta-multitask (코사인 유사도)
- 하이브리드: RRF(Reciprocal Rank Fusion)으로 두 방식 결합

### 결과

| 검색 방식 | Recall@5 | MRR |
|----------|----------|-----|
| 임베딩 (벡터) | **28.0%** | **0.2333** |
| BM25 (키워드) | 20.0% | 0.1800 |
| 하이브리드 (RRF) | 24.0% | 0.2000 |

### 분석
- 가설 방향은 맞음: 임베딩이 BM25를 +8%p 앞섬. 구어체 질문에서 의미 기반 검색의 이점 확인.
- **기대와 다른 결과**: 하이브리드가 임베딩보다 낮음 (28% → 24%). BM25의 노이즈가 RRF 점수를 희석시킨 것으로 보임. 하이브리드가 항상 두 방식보다 낫다는 통념이 법률 도메인 소규모 데이터에서는 성립하지 않음.
- 전체 Recall이 28%로 낮음: 판례 수 부족(129건), 어휘 격차, 청크 전략이 원인으로 가정 → 2주차 실험 대상.

### 다음 액션
청크 전략 변경 / Query Rewriting / 하이브리드 최적화를 2주차에 실험.

## 실험 2: 청크 전략 비교 — 조 단위 vs 조+항 단위 (2주차)

### 가설
조(條) 단위 청크보다 조+항(條+項) 단위로 세분화하면 질문과 더 정밀하게 매칭되어 Recall이 올라갈 것이다.

### 방법
- `ArticleLevelStrategy`: 조 단위 (현행)
- `ArticleParagraphStrategy`: 조 내 항 단위로 분리 (더 세분화)
- 동일한 임베딩 모델, 동일한 평가셋

### 결과

| 청크 전략 | 검색기 | Recall@5 | MRR |
|----------|-------|----------|-----|
| 조 단위 | 임베딩 | 28.0% | 0.2333 |
| 조+항 단위 | 임베딩 | 28.0% | 0.1980 |

### 분석
- **기대와 다른 결과**: 세분화해도 Recall은 동일하고 MRR은 오히려 낮아짐. 항(項) 단위 청크가 너무 짧아 임베딩 품질이 저하되었거나, 조 전체 맥락이 항보다 더 유용할 수 있음.
- 법령 조문은 "조"가 의미 단위로 적합함. 항 하나만으로는 조의 문맥이 빠지므로 정보 손실.

### 다음 액션
조 단위를 유지. 대신 어휘 격차 문제를 Query Rewriting으로 접근.

## 실험 3: Query Rewriting 효과 비교 (2주차)

### 가설
일반인 구어체 질문("전세금 못 받았어요")을 LLM이 법률 용어로 재작성("임대차보증금반환청구")하면 어휘 격차가 줄어들어 검색 성능이 향상될 것이다.

### 방법
- LLM(Claude)이 구어체 질문을 법률 전문 용어로 재작성 후 검색
- 임베딩 단독 vs 하이브리드, 각각 QR 적용 전/후 비교

### 결과

| 조합 | Recall@5 | MRR | vs 베이스라인 |
|------|----------|-----|------------|
| A: 임베딩 / 원문 | 28.0% | 0.2333 | 기준 |
| E: 하이브리드 / 원문 | 24.0% | 0.2000 | -4.0% / -0.0333 |
| F: 임베딩 + QR | 24.0% | 0.1867 | **-4.0%** |
| G: 하이브리드 + QR | **28.0%** | **0.2500** | +4.0% / **+0.0500** |

### 분석
**핵심 발견 — 기대와 다른 결과:**

| 검색기 | QR 효과 | 이유 |
|-------|---------|------|
| 임베딩 단독 | 역효과 (-4%p) | 임베딩 모델은 이미 구어체 의미를 잘 처리함. LLM이 과도하게 법률 용어화하여 오히려 매칭 방해 |
| 하이브리드 | 긍정 (+4%p, MRR +0.05) | QR로 법률 키워드가 명시되면 BM25 성분의 키워드 매칭이 향상됨. RRF에서 BM25 성분을 강화하는 역할 |

**교훈**: QR 효과는 검색기 종류에 따라 다르다. 임베딩에는 불필요, 하이브리드에는 유효. 검색기 조합에 따라 QR 적용 여부를 선택적으로 제어해야 함.

### 다음 액션
- 에이전트 모드에서 하이브리드+QR(G 조합)을 기본 전략으로 채택
- QR LLM 호출 비용이 발생하므로 실서비스에서는 캐싱 필수

## 실험 4: LLM 메타데이터 태깅 효과 (2주차)

### 가설
각 청크에 LLM이 생성한 쉬운 말 요약과 검색 키워드를 추가하면, 임베딩이 구어체 질문과 청크 간 의미 거리를 더 잘 파악하여 검색 성능이 향상될 것이다.

### 방법
- 각 청크에 `[요약] <쉬운 말 설명> [키워드] <법률 키워드>` 텍스트 추가
- 태깅된 청크로 임베딩 인덱스를 새로 빌드 (sampling 100개)
- 태깅 전후 임베딩 검색 성능 비교

### 결과

| 조건 | Recall@5 | MRR |
|------|----------|-----|
| 임베딩 / 원본 청크 | 28.0% | 0.2333 |
| 임베딩 / 태깅 청크 | 28.0% | 0.2300 |

### 분석
- 태깅 효과가 미미. 가능한 원인:
  1. 샘플 태깅(일부 청크만) → 미태깅 청크가 많아 전체 효과 희석
  2. 임베딩 모델이 이미 법률 조문의 의미를 충분히 파악하고 있어 요약 추가가 중복
  3. 태깅된 요약이 임베딩에서 노이즈로 작용
- 전체 청크 태깅 + 더 정교한 태깅 포맷으로 재시도가 필요하나 시간 제약으로 미시행.

## 실험 5: 서빙 최적화 — LRU 임베딩 캐시 (3주차)

### 가설
동일 쿼리가 반복되는 FAQ성 서비스에서 임베딩 벡터를 캐싱하면 응답 속도를 크게 개선할 수 있다.

### 방법
- `LRUEmbeddingCache(max_size=256)` 구현: MD5 해시 키, OrderedDict 기반 LRU 교체
- 캐시 없음 / 캐시 미스 / 캐시 히트 각 10회 측정 (warmup 2회 제외)

### 결과

| 조건 | 평균 Latency | P95 |
|------|------------|-----|
| 캐시 없음 | 96.6ms | 112.5ms |
| 캐시 미스 (첫 호출) | 23.3ms | 116.3ms |
| **캐시 히트 (반복 쿼리)** | **0.02ms** | **0.02ms** |

**속도 향상: 4,830배** (캐시 없음 → 캐시 히트)

### 분석
- 가설 방향 완전히 일치. FAQ성 질문이 반복되는 실제 법률 상담 서비스에서 매우 효과적.
- max_size=256일 때 메모리 overhead는 수 MB 수준으로 무시 가능.
- 캐시 미스 시 23.3ms (캐시 없음 96.6ms보다 빠름): 모델 워밍업 효과.

### 다음 액션
Streamlit 앱의 임베딩 검색에 LRU 캐시 적용. 에이전트 모드에서도 동일하게 사용.

## 실험 6: 서빙 최적화 — 배치 처리 처리량 (3주차)

### 가설
여러 쿼리를 하나의 배치로 묶어 임베딩하면 하나씩 처리하는 것보다 처리량(throughput)이 증가할 것이다.

### 방법
- 동일한 쿼리 셋을 배치 크기 [1, 8, 32, 64]로 나누어 임베딩 처리
- 처리량(queries/sec) 측정

### 결과

| 배치 크기 | 처리량 (q/s) |
|----------|------------|
| 1 | 8.1 |
| 8 | 16.6 |
| 32 | 18.3 |
| **64** | **21.8** |

### 분석
- 가설 방향 일치: 배치 크기 64에서 최고 처리량 (단일 요청 대비 2.7배).
- CPU 환경에서는 GPU 대비 배치 효과가 제한적 (GPU라면 수십~수백 배 이상 차이가 날 수 있음).
- **적용 영역**: 온라인 서빙보다는 인덱스 빌드 등 오프라인 대량 처리에 유효. 온라인에서는 LRU 캐시가 더 실용적.

## 실험 7: 서빙 최적화 — 동시 요청 처리 (3주차)

### 가설
동시 요청 수(workers)를 늘리면 총 처리량이 선형에 가깝게 증가할 것이다.

### 방법
- ThreadPoolExecutor로 동시성 [1, 2, 4, 8]에서 각 32개 임베딩 요청 처리
- 처리량(req/s) 및 평균 응답시간 측정

### 결과

| 동시성 | 처리량 (req/s) | 평균 응답시간 | P95 |
|-------|------------|-----------|-----|
| 1 | 8.97 | 111.4ms | 142.7ms |
| 2 | 9.75 | 203.2ms | 272.8ms |
| 4 | **10.51** | 366.8ms | 532.5ms |
| 8 | 9.73 | 776.3ms | 1027.3ms |

### 분석
- **기대와 다른 결과**: 선형 스케일 없음. 동시성 4 → 8에서 처리량이 오히려 감소.
- 원인: PyTorch CPU 임베딩은 내부적으로 멀티스레드를 이미 사용함. 외부 ThreadPoolExecutor가 추가되면 스레드 경합으로 역효과.
- 결론: CPU 환경에서 임베딩 병렬화는 프로세스 분리(uvicorn multiple workers)가 유효. LLM API 호출은 I/O bound이므로 asyncio 적용이 더 효과적.

## 핵심 발견 (포트폴리오 관점)

### 발견 1: 하이브리드가 임베딩보다 항상 낫지 않다
일반적으로 "하이브리드 = 최선"으로 알려졌지만, 이 프로젝트에서는 RRF가 오히려 임베딩 점수를 희석시켜 성능이 낮아졌다. BM25 노이즈가 적은 법률 도메인 소규모 데이터에서는 임베딩 단독이 더 안정적이었다.

### 발견 2: QR 효과는 검색기 종류에 의존
Query Rewriting은 만능이 아니다. 임베딩 단독(-4%p)에는 역효과지만 하이브리드(+4%p, MRR +0.05)에서는 효과가 있었다. 임베딩 모델은 이미 구어체를 잘 처리하고 있어 QR이 오히려 과도한 법률 용어화로 방해가 됐다.

### 발견 3: LRU 캐시의 압도적 효과
CPU 환경에서 임베딩 최적화 중 가장 효과적인 것은 LRU 캐시였다 (4830배). 복잡한 모델 최적화(ONNX, 양자화) 전에 단순한 캐싱 전략이 실용적임을 확인.

### 발견 4: CPU 동시성 한계
CPU 기반 PyTorch 임베딩에 외부 스레드 동시성을 추가해도 처리량이 선형으로 늘지 않는다. I/O bound LLM 호출과 CPU bound 임베딩을 분리하는 아키텍처(asyncio + 전용 임베딩 프로세스)가 필요하다.

## 한계

1. **낮은 절대 성능 (Recall 28%)**: 판례 수 부족(129건), 평가 QA 셋 크기(25개)로 통계적 신뢰도 제한.
2. **BM25 노이즈 미해결**: RRF 가중치 튜닝 없이 기본값 사용. 가중치 최적화로 하이브리드 개선 여지 있음.
3. **메타데이터 태깅 미완**: 전체 2929개 청크 태깅 미시행 (비용·시간 제약). 전체 태깅 후 재측정 필요.
4. **임베딩 모델 미세조정 미시행**: jhgan/ko-sroberta-multitask는 일반 한국어 모델. 법률 도메인 QA 쌍으로 파인튜닝 시 성능 향상 기대.
