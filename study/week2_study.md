# 2주차 학습 노트: 검색 방식 비교 실험

이 노트는 이 프로젝트의 2주차(검색 비교 실험) 내용을 처음 공부하는 사람도 이해할 수 있게 정리한 것입니다.

---

## 이번 주 목표가 무엇인가?

1주차에서는 "일단 동작하는 RAG"를 만들었습니다.
2주차 목표는 **어떤 검색 방식이 더 좋은지 실험으로 비교**하고, 일반인 질문과 법률 용어 사이의 **어휘 격차** 문제를 해결하는 것입니다.

---

## 핵심 개념 1: 어휘 격차(Lexical Gap)란?

일반인이 묻는 방식:
> "전세금을 못 받았어요. 어떻게 해야 하나요?"

법령 데이터베이스에 저장된 텍스트:
> "임대차보증금반환청구권", "보증금반환의무", "대항력"

같은 내용인데 **단어가 다릅니다.** 이 불일치를 어휘 격차(lexical gap)라고 부릅니다.
검색기가 이 격차를 극복하지 못하면 관련 조문을 찾지 못합니다.

---

## 핵심 개념 2: BM25 vs 임베딩 검색

### BM25 (키워드 기반)

어떻게 동작하나?
- 질문과 문서에서 단어를 추출합니다.
- 질문에 있는 단어가 문서에 얼마나 많이 나오는지 세어 점수를 냅니다.
- TF-IDF처럼 흔한 단어(조사, 어미)는 낮게, 드문 단어는 높게 점수를 줍니다.

장점: 정확한 법률 용어가 질문에 있을 때 매우 잘 찾습니다.
단점: 같은 뜻이라도 다른 단어를 쓰면 찾지 못합니다 (어휘 격차에 취약).

한국어 처리: 형태소 분석기(`kiwipiepy`) 사용.
- "전세금을" → "전세금" (명사만 추출)
- 조사(`을`, `이`, `가`)를 제거해야 제대로 매칭됩니다.

### 임베딩 검색 (의미 기반)

어떻게 동작하나?
- 문장 전체를 하나의 숫자 벡터(384차원 배열)로 변환합니다.
- "의미가 비슷한 문장"은 벡터 공간에서 가까운 위치에 있습니다.
- 질문 벡터와 가장 가까운 문서 벡터를 찾아 반환합니다.

예시:
```
"전세금을 못 받았어요" → [0.12, -0.34, 0.56, ...]
"임대차보증금반환청구"  → [0.13, -0.31, 0.58, ...]  ← 가깝다!
"자동차 수리비 환급"   → [-0.89, 0.42, 0.11, ...]   ← 멀다
```

장점: 다른 단어를 써도 의미가 같으면 찾아줍니다.
단점: 임베딩 모델이 법률 도메인 용어를 잘 이해하지 못할 수 있습니다.
      (일반 한국어로 학습한 모델은 법률 용어 특화가 안 됨)

이 프로젝트 모델: `jhgan/ko-sroberta-multitask` (384차원, 무료, CPU에서 동작)

## 핵심 개념 3: 하이브리드 검색 (RRF)

BM25와 임베딩 각각의 단점을 보완하기 위해 두 결과를 합칩니다.
Reciprocal Rank Fusion (RRF) 공식:

```
score(문서) = 1/(k + 임베딩순위) + 1/(k + BM25순위)
```
- k = 60 (기본값, 상위 순위의 영향력을 부드럽게 함)
- 두 검색기 모두에서 높은 순위를 받은 문서가 높은 점수를 얻음

예시:
```
문서 A: 임베딩 1위, BM25 3위 → 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
문서 B: 임베딩 5위, BM25 1위 → 1/(60+5) + 1/(60+1) = 0.0154 + 0.0164 = 0.0318
문서 C: 임베딩 1위, BM25 미출현 → 1/(60+1) + 1/(60+20) = 0.0164 + 0.0125 = 0.0289
```
→ A > B > C 순서로 반환

장점: 두 검색기의 점수 스케일이 달라도 순위 기반이라 자연스럽게 합산됩니다.

## 핵심 개념 4: Query Rewriting (쿼리 재작성)

어휘 격차를 해결하는 다른 방법: **LLM이 질문을 법률 용어로 바꿔준다.**

```
입력:  "전세금을 못 받았어요. 어떻게 해요?"
         ↓ LLM (Claude/GPT)
출력:  "임대차보증금반환청구권 행사 절차 및 요건"
키워드: ["임대차보증금", "반환청구", "주택임대차보호법"]
```

그런 다음 변환된 쿼리로 검색을 수행합니다.

기대 효과: 법률 데이터베이스의 텍스트와 단어가 더 잘 맞아 검색 정확도 향상.
실제 결과: 항상 좋은 건 아닙니다. LLM이 지나치게 구체적인 용어로 변환해
           오히려 범위가 좁아질 수 있습니다. 실험을 통해 검증해야 합니다.

주의: LLM API를 한 번 더 호출하므로 비용과 latency가 추가됩니다.
캐시(`CachedQueryRewriter`)를 사용해 같은 질문은 재사용합니다.

## 핵심 개념 5: 청크(Chunk) 전략 비교

### 조(條) 단위 (1주차 기본값)

```
제3조(대항력 등) ①임대차는 그 등기가 없는 경우에도 임차인이
주민등록을 마친 때에는 그 다음 날부터 제3자에 대하여 효력이 생긴다.
②임차주택의 양수인은 임대인의 지위를 승계한 것으로 본다.
→ 한 청크로 저장
```

### 조+항(項) 단위 (2주차 실험)

```
제3조①항: "임대차는 그 등기가 없는 경우에도 임차인이 주민등록을
            마친 때에는 그 다음 날부터 제3자에 대하여 효력이 생긴다."
제3조②항: "임차주택의 양수인은 임대인의 지위를 승계한 것으로 본다."
→ 각각 별개 청크로 저장
```

예상 효과: 세분화할수록 관련성 높은 텍스트를 정밀하게 찾을 수 있다.
실제 결과: 청크가 너무 짧으면 맥락이 부족해 임베딩 품질이 떨어질 수 있습니다.
           "짧은 게 항상 좋다"는 직관이 틀릴 수 있음을 보여주는 실험입니다.

## 핵심 개념 6: 평가 지표

### Recall@k

"k개의 검색 결과 중에 정답이 있었는가?" (0 or 1)

예시: k=5일 때 5개 결과 중 정답 조문이 1개라도 있으면 1점, 없으면 0점.
25개 질문 평균: 25개 중 7개 맞으면 Recall@5 = 7/25 = 28%

### MRR (Mean Reciprocal Rank)

정답이 몇 번째에 나왔는지도 반영합니다.

```
1위에 정답 → 1/1 = 1.0
2위에 정답 → 1/2 = 0.5
3위에 정답 → 1/3 = 0.33
없음       → 0
```

여러 질문의 평균값 = MRR.
MRR이 높을수록 정답이 상위에 나오는 검색기입니다.

## 결과 해석 방법

실험 결과 예시:
```
실험                      Recall@k     MRR  vs 베이스라인
-------------------------------------------------------
A-emb/article              28.0%  0.1800
B-emb/para                 24.0%  0.1520          -4.0%
C-bm25/article             20.0%  0.1200          -8.0%
D-bm25/para                16.0%  0.0900         -12.0%
E-hybrid/article           24.0%  0.1600          -4.0%
F-emb+QR/article           32.0%  0.2100          +4.0%
G-hybrid+QR/article        28.0%  0.1900           0.0%
```

이런 결과가 나온다면:
- B가 A보다 낮음 → 법령 조문은 조 전체가 더 좋은 단위. 항이 너무 짧음.
- C가 A보다 낮음 → 구어체 질문에서 임베딩이 유리. 어휘 격차 문제 확인.
- E가 A보다 낮음 → 하이브리드가 항상 좋지는 않음. BM25 노이즈가 RRF 희석.
- F가 A보다 높음 → Query Rewriting 효과 확인. LLM이 도움됨.

## 이번 주 핵심 배움

1. "세분화할수록 좋다"는 직관이 항상 맞지 않습니다. 법령 조문처럼 짧고 압축적인 텍스트는 조(條) 전체가 더 좋은 단위일 수 있습니다.

2. "하이브리드가 항상 더 좋다"는 것도 가설입니다. BM25가 법률 도메인에서 적절한 키워드를 못 찾으면 하이브리드가 임베딩 단독보다 나쁠 수 있습니다.

3. Query Rewriting은 비용이 들지만 어휘 격차를 줄이는 유효한 방법입니다. 단, "항상" 더 좋지는 않습니다.

4. 모든 실험은 숫자로 비교하고, 기대와 다를 때 왜 그런지 분석하는 것이 핵심입니다.
