## 1. 이 프로젝트가 뭘 만드는 건가요?

### 문제 상황

> "집주인이 전세금을 안 돌려줘요. 어떻게 해야 하나요?"

이런 질문을 가진 일반인은 세 가지 문제에 부딪힙니다.

- 법률 전문가에게 상담하려면 **돈이 든다**
- 직접 찾아보려 해도 법조문은 **너무 어렵다** ("임차인은 대항력을 취득한다…")
- 인터넷 검색 결과는 **출처가 불명확**하다

### 해결책

이 프로젝트는 다음을 만듭니다.

```
[사용자 질문 입력]
        ↓
관련 법조문과 판례를 자동 검색
        ↓
AI가 쉽게 설명 + 출처(법 몇 조) 함께 제시
        ↓
"법적 자문이 아니며 전문가 상담 권장" 안내 포함
```

쉽게 말하면: **"법령/판례 DB를 갖춘 AI 상담 도우미"** 입니다.

## 2. RAG란 무엇인가요?

### 일반 AI와 RAG AI의 차이

**일반 AI (GPT, Claude 등)에게 물어보면:**

```
질문: "주택임대차보호법 제3조가 뭐야?"
AI:  "제가 학습한 내용에 따르면... (하지만 법이 개정됐을 수도 있고,
      틀릴 수도 있고, 출처가 없음)"
```

**RAG AI에게 물어보면:**

```
질문: "주택임대차보호법 제3조가 뭐야?"

[내부 과정]
① 법령 DB에서 '주택임대차보호법 제3조' 검색
② 검색된 실제 조문을 AI에게 넘겨줌
③ AI가 그걸 읽고 설명

결과: "주택임대차보호법 제3조에 따르면 (실제 조문 내용)입니다.
       [출처: 주택임대차보호법 제3조, 시행 2023.09.01]"
```

### RAG = Retrieval-Augmented Generation

| 단어 | 뜻 | 이 프로젝트에서 |
|------|-----|----------------|
| Retrieval | 검색(찾아오기) | 법령·판례 DB에서 관련 조문 검색 |
| Augmented | 보강된 | 검색 결과를 AI에게 추가로 제공 |
| Generation | 생성 | AI가 설명 텍스트 생성 |

> **핵심 포인트**: AI가 기억에서 꺼내는 게 아니라, 매번 실제 문서를 검색해서 답변한다.
> 그래서 출처가 명확하고 최신 정보를 반영할 수 있다.

## 3. 데이터 수집 — API와 XML

### 법령·판례 데이터는 어디서 오나요?

국가에서 운영하는 **국가법령정보 공동활용 서비스** (open.law.go.kr)가 있습니다.
여기에 신청하면 모든 법령과 판례를 프로그램으로 가져올 수 있는 열쇠(API 키)를 줍니다.

### API란?

> API = 두 프로그램이 데이터를 주고받는 창구

마치 식당 주문처럼:
- 손님(우리 코드) → 주문서(요청) → 주방(서버) → 음식(데이터)

```
우리 코드가 보내는 요청 예시:
https://www.law.go.kr/DRF/lawSearch.do
  ?OC=내이메일@example.com    ← 신원 확인 (이게 API 인증)
  &target=law                ← 법령을 달라 (prec이면 판례)
  &type=XML                  ← XML 형식으로 달라
  &query=주택임대차보호법      ← 이 법령을 검색해줘
```

이 주소를 브라우저에 치면 서버가 XML 형태로 답해줍니다.

### XML이란?

> XML = 데이터를 태그로 구조화한 텍스트 형식

HTML처럼 꺾쇠 괄호(`<>`)를 씁니다.

```xml
<LawSearch>
  <totalCnt>1</totalCnt>
  <law>
    <법령일련번호>176697</법령일련번호>
    <법령명한글>주택임대차보호법</법령명한글>
    <시행일자>20230901</시행일자>
  </law>
</LawSearch>
```

이걸 보면: "주택임대차보호법의 번호는 176697이고, 2023년 9월 1일 시행"임을 알 수 있습니다.

### 법령 본문 XML

법령 번호(176697)를 알면 실제 조문을 가져올 수 있습니다.

```xml
<법령>
  <기본정보>
    <법령명_한글>주택임대차보호법</법령명_한글>
    <시행일자>20230901</시행일자>
  </기본정보>
  <조문>
    <조문단위>
      <조문번호>3</조문번호>
      <조문제목>대항력 등</조문제목>
      <항>
        <항번호>1</항번호>
        <항내용>임차인은 주택의 인도와 주민등록을 마친 때에는 그 다음 날부터
                제삼자에 대하여 효력이 생긴다.</항내용>
      </항>
    </조문단위>
  </조문>
</법령>
```

### 판례 XML

```xml
<PrecService>
  <판례일련번호>123456</판례일련번호>
  <사건명>임대차보증금반환</사건명>
  <사건번호>2020다12345</사건번호>
  <법원명>대법원</법원명>
  <선고일자>20201201</선고일자>
  <판시사항>임대인이 보증금을 반환하지 않는 경우...</판시사항>
  <판결요지>주택임대차보호법 제3조에 따라...</판결요지>
  <판결내용>원고의 청구를 인용한다...</판결내용>
</PrecService>
```

### 중요한 실수와 교훈

처음에 코드를 짤 때 판례 본문을 가져오는 파라미터를 `MST=`로 썼는데,
실제 API는 `ID=`를 요구했습니다. 그래서 서버가 "없는 판례입니다"라는 HTML을
보내왔고, 이걸 XML인 줄 알고 저장해버렸습니다.

**배운 점**: API를 처음 쓸 때는 반드시 응답이 XML인지 HTML인지 확인해야 합니다.
코드에 아래처럼 감지 함수를 추가했습니다.

```python
def _is_html(text: str) -> bool:
    """API가 오류 HTML을 반환했을 때 감지."""
    stripped = text.lstrip()
    return stripped.startswith("<!DOCTYPE") or stripped.startswith("<html")
```

## 4. 파싱 — 날것의 XML을 정돈하기

### 파싱이란?

> 파싱(Parsing) = 구조화된 텍스트에서 필요한 정보를 꺼내 정리하는 것

XML 파일을 읽어서 파이썬 객체로 변환하는 과정입니다.

```
XML 파일 (서버 응답)          파이썬 객체 (우리가 쓸 형태)
─────────────────────    →    ──────────────────────────
<조문번호>3</조문번호>          StatuteArticle(
<조문제목>대항력</조문제목>          law_name="주택임대차보호법",
<항내용>임차인은...</항내용>         article_no="제3조",
                                  article_title="대항력 등",
                                  content="임차인은..."
                              )
```

### 왜 파싱이 필요한가?

컴퓨터가 XML을 그냥 읽으면 그냥 긴 문자열입니다.
파싱을 통해 "제3조의 내용은 뭐야?" 같은 질문에 바로 답할 수 있는
**구조**를 만들어줍니다.

### 이 프로젝트에서의 데이터 구조

**법령 조문 하나 (`StatuteArticle`)**:
```python
@dataclass
class StatuteArticle:
    law_id: str           # 법령 번호 (예: "176697")
    law_name: str         # 법령 이름 (예: "주택임대차보호법")
    article_no: str       # 조 번호 (예: "제3조")
    article_title: str    # 조 제목 (예: "대항력 등")
    content: str          # 조 전체 내용 텍스트
    enforcement_date: str # 시행일자
    chunks: list[str]     # 항 단위로 잘라놓은 버전 (2주차용)
```

**판례 하나 (`Precedent`)**:
```python
@dataclass
class Precedent:
    prec_id: str          # 판례 번호
    case_name: str        # 사건 이름
    case_no: str          # 사건 번호 (예: "2020다12345")
    court: str            # 법원 이름
    judgment_date: str    # 선고 날짜
    issues: str           # 판시사항 (핵심 쟁점)
    summary: str          # 판결요지 (요약 결론)
    content: str          # 판결내용 (전체)
    cited_statutes: list  # 이 판례가 언급한 법조항들
```

> **`@dataclass`란?** 파이썬에서 데이터를 담는 그릇(클래스)을 쉽게 만드는 문법입니다.
> 엑셀 시트의 한 행(row)과 비슷하게 여러 필드를 묶어서 관리합니다.

## 5. 청킹 — 긴 글을 잘라서 검색에 쓰기

### 왜 잘라야 하나요?

"근로기준법"을 통째로 검색 인덱스에 넣으면 어떻게 될까요?

- 근로기준법은 수백 개의 조로 이루어져 있습니다
- "임금 체불"을 검색했는데 근로기준법 전체가 검색 결과로 나오면 쓸모없습니다
- 정확히 관련된 **조(條)** 하나가 나와야 합니다

그래서 법령을 **조 단위로 잘라서** 각각을 검색 가능한 단위로 만듭니다.

### 청크(Chunk)란?

> 청크 = 검색 인덱스의 최소 단위

```
근로기준법 (법령 전체)
├── 제1조(목적)  ← 청크 1개
├── 제2조(정의)  ← 청크 1개
├── ...
├── 제36조(금품 청산)  ← 청크 1개  ← "임금 체불" 검색 시 이게 나와야 함
└── ...
```

**판례는?** 판결 전체 내용이 길기 때문에:
- `판시사항 + 판결요지`를 하나의 청크로 (핵심 내용)
- `판결내용`이 길면 800자씩 슬라이딩 윈도우로 추가 분할

### 청크 데이터 구조 (`Chunk`)

```python
@dataclass
class Chunk:
    chunk_id: str     # 고유 ID (예: "statute_176697_0003")
    doc_type: str     # "statute" 또는 "precedent"
    source_id: str    # 법령 번호 or 판례 번호
    source_name: str  # 법령명 or 사건명
    text: str         # 실제 검색/임베딩에 쓸 텍스트 내용
    metadata: dict    # 추가 정보 (조번호, 법원명 등)
```

> 최종적으로 모든 법령 조문과 판례는 Chunk 리스트로 변환되어 검색 엔진에 들어갑니다.

## 6. 임베딩 검색 — "의미가 비슷한 걸" 찾는 방법

### 문제: 단어가 달라도 같은 뜻일 때

```
사용자 질문: "전세금을 못 받았어요"
법령 조문:   "임대차보증금반환청구권"
```

이 둘은 단어가 전혀 다르지만 같은 이야기입니다.
키워드 검색으로는 못 찾습니다.

### 해결책: 임베딩(Embedding)

> 임베딩 = 텍스트를 숫자 벡터로 변환하는 것

```
"전세금을 못 받았어요"  →  [0.12, -0.45, 0.87, ...]  (384개 숫자)
"임대차보증금반환"      →  [0.11, -0.43, 0.85, ...]  (384개 숫자)

↑ 숫자들이 비슷하다! = 의미가 비슷하다
```

마치 지도 위의 좌표처럼, 의미가 비슷한 텍스트는 숫자 공간에서 가까이 위치합니다.

### 어떻게 "가깝다"를 측정하나요? — 코사인 유사도

두 벡터가 얼마나 같은 방향을 가리키는지를 측정합니다.
- 1.0 = 완전히 같은 방향 = 내용이 거의 동일
- 0.0 = 수직 방향 = 관련 없음
- -1.0 = 반대 방향 = 반대 의미

```python
# 코사인 유사도 계산 (내적)
scores = self.embeddings @ query_vector
# embeddings: (문서 수 × 384) 행렬
# query_vector: (384,) 벡터
# scores: (문서 수,) — 각 문서와의 유사도 점수
```

### 임베딩 모델: `jhgan/ko-sroberta-multitask`

- 한국어에 특화된 문장 임베딩 모델
- `sentence-transformers` 라이브러리로 사용
- 텍스트 하나를 384차원 벡터로 변환

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("jhgan/ko-sroberta-multitask")
vector = model.encode(["전세금을 못 받았어요"])
# → shape: (1, 384) — 384개 숫자의 배열
```

### 인덱스 구조 (numpy 배열)

```
self.embeddings = shape (N, 384)  ← N개 청크의 임베딩
self.chunks     = [Chunk, Chunk, ...]  ← 청크 목록

검색 시:
1. 질문 → 임베딩 벡터 (1, 384)
2. self.embeddings @ query_vector → (N,) 유사도 점수
3. 점수 내림차순 정렬 → 상위 5개 반환
```

## 7. BM25 검색 — "단어가 같은 걸" 찾는 방법

### BM25란?

> BM25 = 키워드 일치 기반의 고전 검색 알고리즘 (구글 초창기 원리와 유사)

임베딩이 "의미"를 보는 것과 달리, BM25는 **단어(키워드)가 얼마나 자주, 어디에 나오는지**를 봅니다.

### BM25 점수 계산 원리

단순하게 설명하면:
- 질문에 나온 단어가 문서에 있으면 점수 UP
- 그 단어가 전체 문서에서 **희귀할수록** 점수가 더 UP (흔한 단어는 가치가 낮음)
- 문서가 **짧을수록** 해당 단어가 나왔을 때 의미가 있다고 봄

예시:
```
질문: "임금 체불"

문서A: "임금 체불 시 사용자는 3년 이하 징역에 처한다"   → 점수 높음 (두 단어 모두 있음)
문서B: "근로계약서 작성 의무"                           → 점수 낮음 (두 단어 없음)
문서C: "임금은 매월 1회 이상 정기적으로 지급해야 한다"   → 중간 (임금만 있음)
```

### 한국어에서 BM25의 특수 문제 — 형태소 분석

영어: `"wage theft"` → `["wage", "theft"]` 공백으로 나누면 됨

한국어: `"임금체불"` → 공백이 없어서 그냥 자르면 "임금체불" 전체가 하나의 토큰이 됨

그래서 **형태소 분석기(kiwipiepy)**를 씁니다.

```python
from kiwipiepy import Kiwi
kiwi = Kiwi()

tokens = kiwi.tokenize("임금을 체불한 사용자")
# → [("임금", "NNG"), ("체불", "NNG"), ("사용자", "NNG"), ...]
#     ↑단어         ↑품사(일반명사)
```

명사, 동사, 형용사만 추출해서 BM25 인덱스를 만들면,
조사("을", "이", "는")나 어미("하다", "한", "했다")는 무시됩니다.

### 임베딩 vs BM25 — 언제 뭐가 더 좋나?

| 상황 | 임베딩 검색 | BM25 검색 |
|------|------------|-----------|
| "전세금 못 받음" → "보증금반환" | ✅ 의미 파악 | ❌ 단어 다름 |
| "제3조의2" 정확히 검색 | ❌ 숫자 약함 | ✅ 정확 매칭 |
| 일반인 구어체 질문 | ✅ 유리 | ❌ 법률 용어 없음 |
| 법률 전문 용어 검색 | △ 경우에 따라 | ✅ 유리 |

> 2주차에서 이 둘을 섞은 **하이브리드 검색**을 실험할 예정입니다.

## 8. LLM — 실제로 답변을 만드는 AI

### LLM이란?

> LLM(Large Language Model) = GPT, Claude 같은 대형 언어 모델

검색으로 찾아온 법조문·판례를 보고, 사람이 이해할 수 있게 설명해주는 역할입니다.

```
[검색 결과]
주택임대차보호법 제3조의2: "임차인은 보증금 중 일정액을 다른 담보물권자보다
                           우선하여 변제받을 권리가 있다..."

[질문]
"전세금을 못 받았어요"

[LLM이 하는 일]
복잡한 법조문을 읽고 → 쉬운 말로 설명 → 출처와 함께 답변 생성
```

### 왜 OpenAI와 Anthropic 둘 다 지원하나요?

이 프로젝트는 두 회사의 AI를 모두 쓸 수 있도록 설계했습니다.

**추상화(abstraction)** 개념을 씁니다:

```
[BaseLLMClient]  ← 공통 인터페이스 (약속)
     ↑
     ├── OpenAIClient   (GPT-4o-mini 구현)
     └── AnthropicClient (Claude 구현)
```

코드 어디서도 "GPT야, 이거 해줘" "Claude야, 이거 해줘"가 아니라
"AI야, 이거 해줘"만 씁니다. 실제 AI는 `.env` 파일의 설정으로 바꿀 수 있습니다.

```python
# 상위 코드 (RAG 파이프라인)
response = self.llm.complete(messages=[...])  # AI가 뭔지 모름

# .env 파일
LLM_PROVIDER=openai   # 또는 anthropic
```

이렇게 하면 나중에 GPT에서 Claude로 바꿀 때 파이프라인 코드를 손댈 필요가 없습니다.

### 시스템 프롬프트 — AI에게 규칙 주기

```python
SYSTEM_PROMPT = """
당신은 법령과 판례를 검색하여 일반인의 질문에 답변하는 정보 검색 보조 도구입니다.

[중요 규칙]
1. 반드시 제공된 검색 컨텍스트에 근거하여 답변하세요.
2. 답변 마지막에 반드시 출처를 명시하세요.
3. "무조건 승소합니다" 같은 확정적 법률 결론을 단언하지 마세요.
4. "관련 법령에 따르면" 등의 정보 제공 어조를 유지하세요.
"""
```

이게 중요한 이유: AI가 학습한 일반 지식으로 아무 말이나 하는 것을 막고,
반드시 검색된 법조문에 근거하게 강제합니다.

## 9. RAG 파이프라인 전체 흐름

### 한 번의 질문-답변 과정

```
사용자: "전세금을 못 받았어요"
                │
                ▼
┌────────────────────────────────┐
│  1. 임베딩 검색                │
│     "전세금 못 받음" → 벡터    │
│     → 관련 조문/판례 상위 5개  │
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  2. 컨텍스트 구성               │
│  [법령: 주택임대차보호법 제3조의3]│
│  "임차인은 임차주택의..."       │
│                                │
│  [판례: 2020다12345 / 대법원]   │
│  "[판결요지] 보증금반환..."     │
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  3. LLM에 전달                  │
│  "위 법령·판례를 바탕으로       │
│   전세금 못 받은 상황에 답변해" │
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  4. 답변 생성                   │
│  "주택임대차보호법에 따르면...  │
│   [출처] 제3조의3 / 2020다12345│
│   ※ 법률 자문이 아님..."       │
└────────────────────────────────┘
```

### 디스클레이머 강제 삽입

```python
# 법률 책임 설계 — 모든 답변에 반드시 포함
LEGAL_DISCLAIMER = (
    "※ 이 답변은 법령·판례 정보를 검색하여 제공한 참고 정보입니다. "
    "법적 효력이 있는 자문이 아니며, 실제 법적 판단은 변호사 등 "
    "전문가 상담이 필요합니다."
)

# AI가 까먹고 안 넣으면 강제 추가
if "법률 자문" not in answer_text:
    answer_text += f"\n\n{LEGAL_DISCLAIMER}"
```

왜 이게 중요한가: AI가 "무조건 이깁니다"라고 하면 사용자가 잘못된 판단을 할 수 있기 때문입니다.

## 10. 평가용 QA 셋

### "내 시스템이 잘 작동하는지" 어떻게 알까요?

느낌이나 한두 번 테스트로는 알 수 없습니다.
**정답이 있는 문제 세트**를 미리 만들어두고, 시스템이 몇 개 맞히는지 측정합니다.

### 이 프로젝트의 QA 셋 (`data/qa_set.jsonl`)

```json
{
  "id": "q001",
  "question": "전세금을 돌려받지 못하고 있어요. 어떻게 해야 하나요?",
  "answer_type": "statute+precedent",
  "source_statute": "주택임대차보호법 제3조의2, 제3조의3",
  "source_precedent": "임대차보증금반환 관련 판례",
  "difficulty": "medium",
  "domain": "주거/임대차"
}
```

각 질문에 대해 **어떤 법조문이 나와야 하는지**를 미리 정해뒀습니다.

### 평가 지표 — Recall@k

> "정답 조문이 검색 결과 상위 k개 안에 들어있는가?"

```
질문: "전세금을 못 받았어요"
정답 조문: 주택임대차보호법 제3조의3

검색 결과 상위 5개:
  1위: 주택임대차보호법 제3조의3  ← 정답!
  2위: 주택임대차보호법 제8조
  3위: 민법 제621조
  4위: ...
  5위: ...

Recall@5 = 1 (정답이 5위 안에 있음)
Recall@1 = 1 (1위가 정답)
```

여러 질문에 대해 평균을 내면 시스템의 검색 성능을 숫자로 알 수 있습니다.

## 11. 코드 구조 한눈에 보기

```
src/
├── collection/          # 데이터 수집
│   ├── law_api_client.py      → API 요청/응답 담당
│   ├── statute_collector.py   → 법령 XML 파싱
│   └── precedent_collector.py → 판례 XML 파싱
│
├── chunking/            # 청크 분할
│   └── strategies.py          → 법령=조 단위, 판례=판결요지 우선
│
├── retrieval/           # 검색 엔진
│   ├── embedding_retriever.py → 임베딩(의미) 검색
│   └── bm25_retriever.py      → BM25(키워드) 검색
│
├── llm/                 # AI 모델 연결
│   ├── base.py                → 공통 인터페이스 정의
│   └── providers/
│       ├── openai_client.py   → OpenAI(GPT) 연결
│       └── anthropic_client.py → Anthropic(Claude) 연결
│
└── rag/                 # 전체 파이프라인 연결
    └── pipeline.py            → 검색 + LLM 조합 + 디스클레이머

scripts/
├── collect_data.py   → 법령·판례 수집 실행 스크립트
├── build_index.py    → 임베딩 인덱스 빌드 스크립트
└── debug_api_response.py → API 응답 구조 확인용
```

### 전체 실행 순서

```bash
# 1단계: 데이터 수집 (API로 법령·판례 가져오기)
python scripts/collect_data.py
# → data/raw/*.xml (원본)
# → data/processed/statutes.jsonl (정리된 법령)
# → data/processed/precedents.jsonl (정리된 판례)

# 2단계: 임베딩 인덱스 구축
python scripts/build_index.py
# → data/index/embedding_index.npy (검색 인덱스)

# 3단계: 앱 실행
streamlit run app/streamlit_app.py
```
